### **3. POST /auth/refresh**

**Endpoint:** `POST /api/auth/refresh`  
**Handler:** `/backend/internal/handlers/auth/refresh.go`

#### **Request Body**
```json
{
  "refresh_token": "jwt_refresh_token_string"
}
```

✅ **Validaciones:**
- refresh_token obligatorio ✅ (binding:"required")
- Token validado con mismo secret que access token ✅
- Verifica que usuario siga existiendo en DB ✅ (línea 58-64 en refresh.go)

✅ **Lógica de Negocio:**
- Valida firma JWT del refresh token ✅
- Verifica expiración (7 días) ✅
- Consulta DB para verificar usuario activo ✅
- Genera nuevo PAR de tokens (access + refresh) ✅ **Token rotation best practice**
- Invalida refresh token anterior (el nuevo reemplaza al viejo) ✅

✅ **Response (200 OK):**
```json
{
  "access_token": "new_jwt_access_token",
  "refresh_token": "new_jwt_refresh_token"
}
```

✅ **Errores:**
- 400 - Datos inválidos (refresh_token requerido) ✅
- 401 - Refresh token inválido o expirado ✅
- 401 - Usuario no encontrado ✅
- 429 - Rate limit excedido (5 requests cada 15 minutos) ✅

✅ **Security Logging:**
- `auth.refresh.success` → user_id, email, IP ✅
- `auth.refresh.failed` → IP, reason (invalid_token o user_not_found) ✅

**Testing:** ✅ 6/6 tests PASADOS
- Refresh con token válido → HTTP 200 + nuevos tokens ✅
- Nuevo access token funcional → GET /accounts HTTP 200 ✅
- Refresh con token inválido → HTTP 401 ✅
- Refresh sin token → HTTP 400 ✅
- Rate limiting funcionando → HTTP 429 ✅
- Logging verificado en producción ✅

## ⚠️ **OBSERVACIONES MENORES (NO CRÍTICAS)**

### 1. **Refresh Token NO se Guarda en Cookie HttpOnly**

**Descripción:**  
El refresh token se retorna en JSON response, NO en cookie httpOnly (más seguro contra XSS).

**Impacto:** Bajo. Si el frontend guarda el refresh token en localStorage, es vulnerable a XSS. Cookie httpOnly sería más seguro.

**Estado:** ✅ Funcional pero subóptimo

**Recomendación FUTURA:** Agregar handler alternativo `RefreshFromHeader()` que lea el refresh token desde cookie (ya implementado en refresh.go líneas 103-184, pero no activado en router).

---

### 2. **Validación de Password Solo en Frontend**

**Descripción:**  
En **register** se valida `min=8` ✅, pero en **login** NO hay validación mínima (solo `required`).

**Impacto:** Ninguno (es correcto). En login no tiene sentido validar longitud porque estamos comparando contra el hash guardado.

**Conclusión:** ✅ Implementación correcta

---

### 3. **Email Normalizado pero NO Documentado**

**Descripción:**  
El código hace `strings.ToLower(strings.TrimSpace(req.Email))` en register (línea 60) y login (línea 47), pero la documentación NO menciona esto.

**Impacto:** Bajo. Es una buena práctica que evita problemas de "User@Example.com" vs "user@example.com".

**Recomendación:** Agregar a API.md una nota: "Nota: Los emails se normalizan automáticamente a minúsculas".

---

### 4. **Config Fallbacks Hardcodeados**

**Descripción:**  
Si `JWTAccessExpiry` o `JWTRefreshExpiry` no se parsean desde env vars, hay fallbacks:
```go
accessTokenExpiry = 15 * time.Minute    // Fallback
refreshTokenExpiry = 7 * 24 * time.Hour // Fallback
```

**Impacto:** Ninguno. Es una EXCELENTE práctica de defensive programming.

**Conclusión:** ✅ Correcto

---

### 5. **Updated_at NO se Actualiza Automáticamente**

**Descripción:**  
La tabla `users` tiene campo `updated_at`, pero NO hay trigger en PostgreSQL para actualizarlo automáticamente en cada UPDATE.

**Impacto:** Bajo actualmente, porque NO existe endpoint `PUT /users/:id` para actualizar email/name/password.

**Estado:** ❌ Campo existe pero sin utilidad real todavía

**Recomendación:** Si se implementa funcionalidad de "editar perfil", agregar trigger:
```sql
CREATE TRIGGER update_users_updated_at
BEFORE UPDATE ON users
FOR EACH ROW
EXECUTE FUNCTION update_updated_at_column();
```

## 🐛 **BUGS POTENCIALES ENCONTRADOS**

### ❌ **BUG 1: Inyección SQL Potencial (FALSA ALARMA - ESTÁ BIEN)**

**Descripción:**  
Revisé si había riesgo de SQL injection en:
```go
checkQuery := "SELECT EXISTS(SELECT 1 FROM users WHERE email = $1)"
err := h.db.Pool.QueryRow(ctx, checkQuery, req.Email).Scan(&exists)
```

**Conclusión:** ✅ **NO hay bug**. Se usa parámetro `$1` (prepared statement de pgx), que es inmune a SQL injection.

---

### ⚠️ **POSIBLE BUG 2: No Hay Rate Limiting**

**Descripción:**  
Un atacante puede hacer brute-force a `/auth/login` sin límite de intentos.

**Impacto:** Seguridad. Sin rate limiting, alguien podría probar millones de contraseñas.

**Estado:** ❌ NO implementado

**Recomendación:** Implementar middleware de rate limiting (ej: `github.com/ulule/limiter`) con límite de 5 intentos por IP cada 15 minutos.
